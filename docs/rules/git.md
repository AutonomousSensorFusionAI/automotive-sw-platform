# GIT 사용 규칙

## Commit

1. 커밋은 반드시 작은 단위로 구성되어야 합니다. 예를 들어 한가지의 기능을 넣기 위해 총 3개의 모듈을 수정해야 된다면, 모듈별로 나눠서 커밋을 해도 되고, 모듈 내에서도 함수별로 나눠서 커밋을 해도 됩니다. 작은 커밋은 동료가 더 쉽게 코드를 이해할 수 있도록 도와줍니다. **즉 자주 커밋 해주세요.**
2. 반드시 완료후에 커밋 해주세요. 개발 중에 단순 저장용으로 커밋을 하면 안됩니다. 만약에 정말로 필요에 의해 저장용도로 커밋을 하고 싶다면, 브랜치를 새로 작성해서 해주시고 브랜치를 삭제하는 방법도 있습니다. **그러나 저장 목적으로 커밋은 가급적 하지 말아주세요.**
3. 2번과 연계하여, 완료라는 의미는 변경된 코드에 대해 충분한 테스트가 이루어졌다는 의미입니다. **테스트를 다 하고 커밋해주세요.**
4. 커밋 메세지를 적을 때 가장 어려움을 겪는 것은 영문으로 어떻게 써야될지 모르겠다는 부분입니다. 저희는 한국 사람입니다. 한글로 커밋을 적어도 상관 없습니다. **언어 상관 없이 명확하게 요약하여 해당 커밋의 목적을 분명히 해주세요.**
5. 커밋 메세지에 대한 룰은 아래에 별도로 작성하겠습니다.

## Branch and Pull Request

1. 브랜치는 커밋의 집합입니다. 특정 문제 해결 및 기능 추가를 해결하기 위해 여러가지 코드를 수정하여 완료가 되었다면 그 내용을 묶어서 브랜치로 만들고 PR 과정에 들어가면 됩니다.
2. 리뷰어는 본인의 상급자를 지정 하는것이 아니라, 내 작업에 영향을 받을만한 사람들을 지정하는 과정입니다.
3. 모든 리뷰어의 승인이 있은 후에 Merge 하면됩니다. 리뷰 과정에서의 의견 충돌은 어떤 방법이든 **잘 해결 하세요.**
4. 브랜치를 Merge 해야할 일이 있을 수 있습니다. Merge시에 추천은 Merge Commit을 새로 생성 하는 것입니다. **반드시 필요한 것이 아니라면 Rebase and Merge는 지양해주세요.** 어느 시점에 머지되고 삭제되었는지 알수가 없는 등 문제가 많습니다.
5. 브랜치 네이밍 규칙 및 PR 메세지에 대한 룰은 아래에 별도로 작성하겠습니다.

## Naming Conventions

### Commit

커밋은 제목과 본문으로 이루어져 있습니다. 제목과 본문은 반드시 **한줄 띄어야** 하며, **제목의 길이는 가급적 50글자를 넘지 않도록 합시다.** 또한 커밋의 제목을 보고 해당 커밋이 어떤 역할인지를 쉽게 분류할 수 있도록 제목에 Prefix를 공통으로 지정합니다. 자체 룰이 엄격하지 않기 때문에, CMF(커밋 메세지 포맷터) 등을 활용하는 것은 각자의 자유입니다. 만약 간단한 수정이라 자세한 내용이 불필요하다면 **본문을 제외**해도 됩니다.

- 제목 Prefix 규칙

| Prefix | Description |
| --- | --- |
| FEAT | 새로운 기능 추가 |
| FIX | 버그 수정 |
| DOCS | 문서 수정 및 추가 |
| STYLE | 코드 스타일 관련 변경 (코드 포맷팅, 세미콜론 누락 등) |
| REFACTOR | 코드 리팩토링 |
| TEST | 테스트 코드, 리팩토링 테스트 코드 추가 |
| CHORE | 빌드 Task 수정, 패키지 매니저 수정 (.gitignore 수정, package.json 수정 등) |
- 예시
    
    > FEAT: WebServer 포트 변경기능 추가
    
    기존에 80번으로 고정되어 있던 웹서버의 포트를 서버 시작 시점에 환경설정을 통해 변경 가능하도록 config.json에 boundPort라는 키를 추가하고 Bootstrap 시점에 해당 키를 읽어와 웹서버의 포트를 변경할 수 있게 하는 기능 추가하였음
    > 

### Branch

1. 브랜치 이름은 모두 영문 소문자로 해주세요. 구분자는 언더바(_)를 쓰지말고 하이픈(-)으로 구분하는 것으로 하겠습니다.
- 네이밍 규칙

| Prefix | Description |
| --- | --- |
| feature/ | 새로운 기능 추가 |
| fix/ | 버그 수정 |
| hotfix/ | 운영상 문제로 리뷰 무시하고 수정이 진행되야 되는 급한 수정 |
- 예시

> feature/webserver-port-configurable
> 

### Tag

1. 태그는 아래 형태로 v를 포함하여 진행합니다.
    
    v1.2.3
    

### PR / Issue

1. PR 및 Issue는 모두 템플릿으로 등록해두고 해당 템플릿을 사용하는것으로 하겠습니다.
```
.github/PULL_REQUEST_TEMPLATE.md
.github/ISSUE_TEMPLATE/bug_report.md
.github/ISSUE_TEMPLATE/enhancement_request.md
```
